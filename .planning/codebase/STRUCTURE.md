## Directory Purposes

**spirit-room-full/ (Root):**
- Purpose: Project root containing all infrastructure and management components
- Contains: Two main subsystems (spirit-room core, spirit-room-manager AI)
- Key files: `CLAUDE.md` provides overview and architecture context

**spirit-room/ (Core Infrastructure):**
- Purpose: Docker-based training room infrastructure and host CLI
- Contains: Dockerfile, entrypoint, training scripts, CLI interface
- Key files: `spirit-room` (CLI entry point), `base/Dockerfile` (image definition), `build-base.sh` (build trigger)

**spirit-room/base/ (Docker Build Context):**
- Purpose: Source files and configuration for base Docker image
- Contains: Dockerfile layers, startup scripts, default catalog
- Key files: `Dockerfile` (image definition), `entrypoint.sh` (container init), `catalog.md` (tool reference)

**spirit-room/base/scripts/ (Container Scripts):**
- Purpose: Helper scripts bundled into Docker image for training execution
- Contains: Training automation, progress inspection, configuration templates
- Key files: `start-training.sh` (main training loop), `status.sh` (progress display), `MISSION.md.template` (user task template)

**spirit-room-manager/ (Management AI):**
- Purpose: AI agent persona and workflow for user interaction and room creation
- Contains: Claude instructions, hiring skill, project state tracking
- Key files: `CLAUDE.md` (persona definition), `skills/MR_POPO.md` (hiring workflow)

**.planning/codebase/ (Analysis Artifacts):**
- Purpose: Generated analysis documents for other GSD commands
- Contains: Architecture, structure, conventions, testing, concerns docs
- Generated by: `/gsd-map-codebase` command

## Key File Locations

**Entry Points:**
- `spirit-room`: Host CLI script - user interaction point for room lifecycle (`open`, `enter`, `list`, `close`, `logs`, `auth`)
- `base/Dockerfile`: Base image definition - build artifact for all training containers
- `base/entrypoint.sh`: Container startup orchestration - runs on every container creation
- `spirit-room-manager/CLAUDE.md`: Management AI initialization - read by Claude at startup in manager context
- `base/scripts/start-training.sh`: Training automation - invoked inside container by user via `start-training` command

**Configuration:**
- `base/Dockerfile`: Layer-by-layer image specification (packages, CLIs, credentials setup)
- `base/entrypoint.sh`: Container behavior at startup (services, tmux, status display)
- `base/catalog.md`: Default tool reference (fallback if project doesn't provide override)
- `base/scripts/MISSION.md.template`: Template for user task definition

**Core Logic:**
- `spirit-room`: Container lifecycle management (folder→container mapping, port allocation, SSH setup)
- `base/scripts/start-training.sh`: Two-phase training loop with idempotent flags
- `spirit-room-manager/skills/MR_POPO.md`: Hiring workflow (interview → MISSION.md generation → room launch)

**Testing:**
- No automated test suite present; manual testing via `spirit-room open [test-folder]`

**Documentation:**
- `CLAUDE.md` (project overview): Architecture, repository structure, CLI commands, container layout
- `spirit-room/README.md`: Setup instructions, usage examples, build timing decisions
- `spirit-room-manager/HANDOVER.md`: Implementation status, design decisions, pending features
- `spirit-room-manager/skills/MR_POPO.md`: Hiring workflow steps, MISSION.md generation rules, room launch procedure

## Naming Conventions

**Files:**
- Executable scripts: lowercase with hyphens (`spirit-room`, `build-base.sh`, `start-training.sh`, `status.sh`)
- Documentation: UPPERCASE.md (`CLAUDE.md`, `README.md`, `HANDOVER.md`, `MISSION.md.template`)
- Configuration: lowercase with extension matching purpose (`Dockerfile`, `catalog.md`)
- Templates: `.template` suffix for user-editable template files (`MISSION.md.template`)

**Directories:**
- Core infrastructure: single word or hyphenated lowercase (`spirit-room`, `base`, `scripts`)
- Functional areas: semantic English words (`workspace`, `skills`, `codebase`)
- Generated: leading dot for hidden system directories (`.planning`, `.logs`, `.prepared`, `.done`)

**Variables & Identifiers (in scripts):**
- Bash variables: UPPERCASE for constants/env vars (`BASE_IMAGE`, `AUTH_VOLUME`, `ROOM_NAME`)
- Bash functions: lowercase with underscores (`cmd_open`, `folder_to_name`, `find_free_port`)
- Container names: `spirit-room-{sanitized-folder-name}` (lowercase, sanitized)

## Where to Add New Code

**New Feature (Training Automation):**
- Primary code: `base/scripts/` (shell scripts bundled in image) or modify `base/scripts/start-training.sh`
- Tests: Manual testing via `spirit-room open` with test MISSION.md
- Config: Update `base/Dockerfile` if new tools/packages needed
- Documentation: Update `base/catalog.md` if exposing new tools to agents

**New CLI Command:**
- Implementation: Add `cmd_*` function in `spirit-room` script
- Pattern: Follow existing functions (`cmd_open`, `cmd_enter`, `cmd_list`, etc.)
- Usage section: Update `usage()` function with new command documentation

**New Management Workflow (e.g., monitoring):**
- Skill file: Create `spirit-room-manager/skills/NEW_SKILL.md` with workflow steps
- Invocation: Reference in `spirit-room-manager/CLAUDE.md` instructional section
- Example: Planned `spirit-room monitor start/open` would add monitoring skill

**New Tool/Package:**
- Build image: Add to Dockerfile layers (group by tool category - layer number indicates sequence)
- Catalog reference: Update `base/catalog.md` with tool name, use case, invocation syntax
- Project override: Users create `project-folder/catalog.md` to customize tool availability per training

**New User-Facing Documentation:**
- Quick start: Add to `spirit-room/README.md` or `CLAUDE.md`
- Room operation: Add to `base/scripts/MISSION.md.template` as example or constraint
- Implementation guidance: Update `spirit-room-manager/skills/MR_POPO.md` with new interview questions/MISSION rules

## Special Directories

**base/scripts/ (Bundled in Image):**
- Purpose: Container helper commands available at runtime
- Generated: No (manually authored)
- Committed: Yes (part of source tree)
- Symlinked at runtime: `start-training` and `status` available as global commands in container

**/workspace/ (Project Mount):**
- Purpose: Shared mount between host and container
- Generated: Created per project at user's folder location
- Committed: No (user-specific content)
- Contains: Project code, MISSION.md, catalog.md override, generated .logs/

**/room/ (Image Read-Only):**
- Purpose: Default scripts and catalog bundled in Docker image
- Generated: No (copied from `base/scripts/` and `base/catalog.md` at build time)
- Committed: No (derived from build)
- Contains: Default `catalog.md`, scripts from `base/scripts/` directory

**.logs/ (Generated at Runtime):**
- Purpose: Training progress logging
- Generated: Yes (created by `start-training.sh` on first run)
- Committed: No (local logs, ephemeral)
- Contains: `progress.log` with timestamped training output

**.prepared / .done (Idempotency Flags):**
- Purpose: Signal phase completion for resume capability
- Generated: Yes (created by training agent)
- Committed: No (ephemeral state flags)
- Used by: `start-training.sh` to skip completed phases on restart

**docker volumes (Named, Persistent):**
- `spirit-room-auth`: Claude Code authentication (shared across rooms)
- `spirit-room-opencode-auth`: opencode authentication (shared across rooms)
- Committed: No (data volumes, created at runtime)
