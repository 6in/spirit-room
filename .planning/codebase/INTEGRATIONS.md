# External Integrations

**Analysis Date:** 2026-04-13

## APIs & External Services

**AI Code Generation:**
- Claude Code CLI - Anthropic AI code generation service
  - SDK/Client: `@anthropic-ai/claude-code` npm package
  - Auth: Device Flow login via `claude auth login` command
  - Location: Invoked in `spirit-room/base/scripts/start-training.sh` line 27
  - Scope: Full file operations, code generation, editing, refactoring

- opencode AI - Multi-provider AI coding platform
  - SDK/Client: `opencode-ai` npm package
  - Auth: `opencode auth` command for provider configuration
  - Location: Invoked in `spirit-room/base/scripts/start-training.sh` line 28
  - Scope: Alternative execution engine, multi-provider support

## Data Storage

**Databases:**

- SQLite3 - Local embedded database
  - Pre-installed in base image at `spirit-room/base/Dockerfile` line 16
  - No external connection needed
  - Used for: Local POC data storage

- Redis - In-memory cache/data structure store
  - Pre-installed in base image at `spirit-room/base/Dockerfile` line 17
  - Auto-started via `entrypoint.sh` line 15
  - Accessible on localhost:6379 within container
  - Used for: Caching, session management, inter-process communication

**Persistent Volumes:**
- Docker named volume: `spirit-room-auth`
  - Mount path: `/root/.claude` inside container
  - Purpose: Stores Claude Code authentication credentials (persistent across container restarts)
  - Shared across all training rooms

- Docker named volume: `spirit-room-opencode-auth`
  - Mount path: `/root/.config/opencode` inside container
  - Purpose: Stores opencode authentication configuration
  - Shared across all training rooms

**File Storage:**
- Host filesystem via Docker bind mount
  - Host path: User-specified project folder (e.g., `~/projects/langgraph-poc`)
  - Container path: `/workspace`
  - Read-write access for all POC outputs

- Log storage: `/workspace/.logs/progress.log`
  - Generated by `start-training.sh` at line 5
  - Persisted on host filesystem
  - Tailed in real-time via tmux pane in `entrypoint.sh` line 47

## Authentication & Identity

**Auth Provider:**
- Anthropic Claude Code - Device Flow authentication
  - Implementation: Interactive browser-based login via `claude auth login`
  - Credentials stored in: Docker volume `spirit-room-auth` at `/root/.claude`
  - Established once per host via `spirit-room auth` command
  - Shared across all container rooms without re-authentication
  - Status check: `claude auth status` in `status.sh` line 28

- opencode - Multi-provider auth management
  - Implementation: Configured via `opencode auth` command
  - Credentials stored in: Docker volume `spirit-room-opencode-auth` at `/root/.config/opencode`
  - Optional - only required if using opencode engine instead of Claude Code

**Access Control:**
- SSH container access
  - Default credentials: root user, password `spiritroom`
  - Configured in `Dockerfile` line 37-39
  - Used for: Remote SSH access via `spirit-room enter` command
  - Port: Dynamically assigned starting from 2222 (determined by `find_free_port` in `spirit-room` line 41-47)

## CI/CD & Deployment

**Hosting:**
- Docker containers as ephemeral training environments
- No persistent deployment - containers stopped/removed after training
- Workspace preserved on host after `spirit-room close`

**Container Management:**
- Docker daemon on host machine
- Manual container lifecycle via `spirit-room` CLI:
  - `spirit-room open` - Launches new container
  - `spirit-room list` - Views active containers
  - `spirit-room close` - Stops and removes container
  - All operations performed via Docker CLI commands in `spirit-room` script

**Build Process:**
- Image building: `./spirit-room/build-base.sh`
  - Builds `spirit-room-base:latest` image
  - Docker build output: `--progress=plain` for detailed logs
  - Requires rebuild when tool versions change (documented in `README.md` lines 112-123)
