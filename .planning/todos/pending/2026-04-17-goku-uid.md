## Problem

精神と時の部屋のコンテナは現在 `root` で実行されており、`/workspace` はホストフォルダをマウントしている。修行中に作成・更新されたファイルはすべて `root:root` 所有になるため、コンテナ終了後にホスト側で `sudo chown -R` を毎回実行しなければ編集や削除ができず面倒。

該当箇所:
- `spirit-room/base/Dockerfile:54` で `root:spiritroom` を設定し SSH も root ログイン許可
- `spirit-room/base/Dockerfile:82` で `WORKDIR /workspace`（root 実行）
- `spirit-room/spirit-room:77-86` の `docker run` でユーザー指定なし
- `spirit-room/base/CLAUDE.md` でも `/workspace` への成果物書き込みを規約化

## Solution

コンテナ内に `goku` ユーザーを作成し、ホストユーザーと UID/GID を一致させる:

1. **Dockerfile**: ビルド時 ARG ではなく実行時に作成する方が柔軟（ホストごとに UID が違うため）。`entrypoint.sh` で以下を実行:
   - `HOST_UID` / `HOST_GID` 環境変数を受け取る
   - `groupadd -g $HOST_GID goku` / `useradd -m -u $HOST_UID -g $HOST_GID -s /bin/bash goku`
   - `echo "goku ALL=(ALL) NOPASSWD:ALL" >> /etc/sudoers.d/goku`
   - 既存の `/root/.claude` 等の認証ボリュームを `goku` でも読めるようにシンボリックリンク or chown
2. **spirit-room CLI**: `docker run` に `-e HOST_UID=$(id -u) -e HOST_GID=$(id -g)` を追加
3. **SSH/tmux**: `cmd_enter` で `ssh goku@localhost` に切り替え。`tmux` セッションも goku で起動
4. **claude/opencode 認証**: 共有ボリューム `spirit-room-auth` の所有者を goku に合わせる必要あり（既存ボリュームのマイグレーションも考慮）
5. **検証**: 修行終了後にホスト側で `ls -la` して所有権が自分のユーザーになっているか確認

懸念点:
- 既存 `spirit-room-auth` ボリュームに root 所有のファイルがあるため、初回起動時に chown が必要
- `kaio` モード (`cmd_kaio`) でも同じ対応が必要
- `sudo` が必要なシーン（apt install など）が修行スクリプトにある場合、`sudo` 経由に書き換える必要あり
