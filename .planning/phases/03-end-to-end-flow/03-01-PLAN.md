       ```bash
       sudo chmod +x /usr/local/bin/spirit-room
       ```

    2. **~/projects/ ディレクトリを作成:**
       ```bash
       mkdir -p ~/projects
       ```

    3. **entrypoint.sh を修正:** `spirit-room/base/entrypoint.sh` の tmux training ペインに送る send-keys 部分（現在: `"echo '部屋[$ROOM_NAME] 準備完了 | start-training で修行開始 | status で確認'" C-m`）を、RESEARCH の Pattern 1 に従って条件分岐に置き換える:
       ```bash
       if [ -f /workspace/MISSION.md ] && [ ! -f /workspace/.done ]; then
           tmux send-keys -t "$SESSION:training" "start-training" C-m
       else
           tmux send-keys -t "$SESSION:training" \
               "echo '部屋[$ROOM_NAME] 準備完了 | start-training で修行開始 | status で確認'" C-m
       fi
       ```
       `.done` が存在する部屋を再起動した場合は自動起動しないこと（多重実行回避）。

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `/usr/local/bin/spirit-room` | accept | 単一ユーザー（本人）環境のローカル開発ツール。sudo 必須のインストールで意図しない改変は想定低 |
| T-03-02 | Elevation of Privilege | entrypoint.sh の自動 start-training | mitigate | `.done` フラグ確認により、既に完了済み部屋での意図しない再実行を防止 |
| T-03-03 | Denial of Service | entrypoint.sh の無限ループ（.prepared が作成されない） | mitigate | start-training.sh 側に既存の sleep ベース再試行実装あり（Phase 2 検証済み）。本プランはそれ以上の変更なし |
| T-03-04 | Information Disclosure | catalog.md の `--break-system-packages` 推奨 | accept | コンテナ内 root 実行のみに影響。システムパッケージ破壊リスクはコンテナ再作成で復旧可能 |
| T-03-05 | Tampering | Docker イメージ再ビルド（供給チェーン） | accept | ローカルビルドのみ、上流は Ubuntu 24.04 公式と npm 公式。Phase 1 で同等リスクを受容済み |

すべて HIGH 以下。BLOCKING な未緩和脅威なし。
