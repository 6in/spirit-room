cmd_auth() {

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host ブラウザ → Anthropic OAuth | ユーザーがホストブラウザで OAuth PKCE フローを完了 |
| ホスト → Docker daemon | `docker run -it` によるインタラクティブセッション |
| Docker ボリューム → コンテナ `/root/.claude` | 認証情報の読み書き |
| 部屋 A ↔ 部屋 B | 同じ名前付きボリュームを介した認証共有 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Information Disclosure | `.credentials.json` に OAuth トークンを平文保存 | accept | Docker 名前付きボリュームは Docker daemon 権限（デフォルトでは root）でのみ読めるホストパス。ローカル開発マシン前提の設計であり、プロジェクト Out of Scope に多ユーザーホスティングは含まれない。 |
| T-02-07 | Spoofing | 悪意あるコンテナが `spirit-room-auth` ボリュームをマウントして認証情報を盗む | accept | 同一 Docker daemon 上の任意コンテナはボリュームにアクセス可能。これはローカル開発者 1 名の前提で受容。Docker ソケットへのアクセスを持つプロセスは既にホストを完全制御できる。 |
| T-02-08 | Tampering | 認証フロー途中でブラウザ中間者がトークンをすり替え | mitigate | `claude auth login` は OAuth PKCE（Proof Key for Code Exchange）を使用しており、トークン引き換えは発行元クライアントのみ可能（Claude Code CLI 組み込み）。追加対策は不要。 |
| T-02-09 | Elevation of Privilege | `docker run -it` でのインタラクティブ実行時の権限昇格 | accept | 一時コンテナ `--rm` で削除。ホスト側はユーザー権限の docker CLI 実行のみ。 |
| T-02-10 | Denial of Service | 同時に複数の `spirit-room auth` を実行しボリュームをロック | mitigate | 実運用では初回 1 回のみ実行するコマンド。並行実行はドキュメントで非推奨とする。 |
</threat_model>
