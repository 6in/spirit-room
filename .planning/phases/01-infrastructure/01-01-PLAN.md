Make exactly two edits to `spirit-room/base/Dockerfile`:

EDIT 1 — Fix BUG-1 (catalog COPY path). Replace line 47:
```
COPY catalog/catalog.md     /room/catalog.md
```
with:
```
COPY catalog.md             /room/catalog.md
```
The file lives at `spirit-room/base/catalog.md` and the build context is `./base`, so the relative path is `catalog.md` (no subdirectory).

EDIT 2 — Fix BUG-3 (sshd_config hardening). Replace the SSH config RUN block (lines 36-39):
```
RUN mkdir /var/run/sshd \
    && echo 'root:spiritroom' | chpasswd \
    && sed -i 's/#PermitRootLogin prohibit-password/PermitRootLogin yes/' /etc/ssh/sshd_config \
    && sed -i 's/#PasswordAuthentication yes/PasswordAuthentication yes/' /etc/ssh/sshd_config
```
with:
```
RUN mkdir /var/run/sshd \
    && echo 'root:spiritroom' | chpasswd \
    && echo "PermitRootLogin yes" >> /etc/ssh/sshd_config \
    && echo "PasswordAuthentication yes" >> /etc/ssh/sshd_config
```
Reason: Ubuntu 24.04's sshd_config defaults may not match the commented patterns the original sed expected, in which case sed exits 0 with no changes and SSH silently rejects root login. OpenSSH applies the last matching directive, so appending is order-safe regardless of pre-existing values.

Do NOT change any other line in the Dockerfile. Do NOT reorder layers.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host filesystem → Docker build context | Files copied into image via COPY become part of any container later started from it |
| Docker image → containers | Anything baked into the image (credentials, configs, packages) is inherited by every running room |
| Network → SSH on host port (later plan) | Plan 02 maps a host port to container :22 — the patched sshd_config governs who can log in |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | sshd password auth (root:spiritroom baked into image) | accept | Local-only sandbox per CLAUDE.md; image is for isolated training rooms on a developer machine, not exposed to untrusted networks. Documented in research as the intended design. Plan 02 will only bind SSH to localhost via `-p {port}:22` (Docker default binds to 0.0.0.0 — note for Phase 2 hardening backlog). |
| T-01-02 | Tampering | Dockerfile COPY path | mitigate | Task 1 hardcodes the exact COPY path `catalog.md /room/catalog.md` and the acceptance criteria greps for both presence of the fix and absence of the broken pattern. |
| T-01-03 | Information Disclosure | sshd config silently misconfigured (BUG-3) leaking false sense of security | mitigate | Task 1 replaces fragile sed with deterministic `echo >>` appends; Task 2 acceptance criteria runs the image and greps `/etc/ssh/sshd_config` to confirm both directives are present in the built image, not just the source. |
| T-01-04 | Denial of Service | apt/npm install layer failures | mitigate | Task 2 instructs the executor to surface build failures rather than fix-forward, preventing partial-image artifacts. Acceptance criteria probes each required CLI inside the built image. |
| T-01-05 | Elevation of Privilege | `chmod +x` on host CLI | accept | The host CLI is owned by the developer; making it executable does not change ownership or grant new privileges. Standard developer-tool hygiene. |
| T-01-06 | Repudiation | No build provenance | accept | Out of scope for v1 sandbox; image is built locally and not distributed. |
</threat_model>
