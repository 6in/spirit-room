    'echo SSH_OK && redis-cli ping && tmux list-windows -t spirit-room' 2>&1 | tee -a "$EVIDENCE"

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host loopback → container SSH | `docker run -p {port}:22` exposes port 22 of the container; verification connects from the host as root with the baked-in password |
| host filesystem → container /workspace | `/tmp/spirit-room-test` is bind-mounted read-write into the container — anything the container does there persists on the host |
| auth volumes → container | `spirit-room-auth` and `spirit-room-opencode-auth` named volumes mount into the container (empty in this phase since auth happens in Phase 2) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-07 | Spoofing | SSH password auth on a published port | accept | Local sandbox only. Docker by default binds `-p 2222:22` to `0.0.0.0` which exposes the port to the LAN; this is documented as a Phase 2/backlog hardening item (bind to `127.0.0.1` explicitly). For Plan 02 verification, the developer machine is the only consumer and verification is short-lived (Task 3 tears down). Acceptable risk for v1 sandbox per CLAUDE.md tech-stack constraints. |
| T-01-08 | Tampering | bind-mounted `/tmp/spirit-room-test` | mitigate | Test directory is created in `/tmp` (developer-owned), populated with nothing sensitive, and removed in Task 3. |
| T-01-09 | Information Disclosure | runtime-evidence.txt contains SSH port and container details | accept | File lives inside `.planning/` which is the project's planning directory; no credentials are written to it (password is passed via sshpass argv, not logged). |
| T-01-10 | Denial of Service | Leftover test container holding port | mitigate | Task 1 force-removes any prior `spirit-room-spirit-room-test` before starting; Task 3 tears down via `spirit-room close` unconditionally. |
| T-01-11 | Repudiation | Manual checkpoint outcome not logged | mitigate | Checkpoint resume-signal requires the human to either type "approved" or describe the failure; the SUMMARY file (output) records the result. |
| T-01-12 | Elevation of Privilege | Container runs as root with SSH password auth | accept | Inherent to the design (training rooms need root for apt/npm); contained by Docker namespace isolation. |
</threat_model>
